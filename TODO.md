# UI Test Backlog

Backlog for the UI e2e phase of the Playwright framework. Agreed 2026-04-24, before concluding UI and moving on to API tests.

## Ground rule
Do not mark an entry as [x] until a test run has confirmed the code passes. "Written" is not "done." If a TC fails, update the entry with the failure and next step rather than ticking it. Learned on 2026-04-24 after otp-login TC12 was prematurely ticked and then had to be reverted.

## Goal
Portfolio-style example framework demonstrating core Playwright concepts — something that can be consulted as a reference in future sessions and shown as a skills portfolio.

## Pending new specs

Each spec below targets a Playwright concept not yet demonstrated. Recommended order: top-down.

- [x] **file-upload.spec.ts** (`/upload`) — `setInputFiles()`, size BVA (499/500/501 KB), recovery after size error and after native required validation. 8 TCs. Single-file only (input has `multiple=false`); no type filter (`accept=""`); drag-and-drop skipped (no custom drop zone). Fixture files generated at runtime into `test-artifacts/upload/` via `Buffer.alloc(N * 1024)`. Added 2026-04-24.
- [x] **file-download.spec.ts** (`/download`) — `Promise.all` + `page.waitForEvent('download')`, `download.suggestedFilename()`, `download.saveAs()` to `testInfo.outputDir` (per-(spec, test, project) isolation, kept-on-failure), text/JSON/PNG/JPEG content + magic-byte assertions, `download.url()` + `download.failure()` page-level invariants looped across all stable files, `page.on('download')` listener pattern. 7 TCs. Targets only the 5 stable demo files (`some-file.txt`, `some-file.json`, `wdio.png`, `cdct.jpg`, `xpath-css.png`) — timestamped user-uploaded files on the page rotate and are excluded as non-deterministic. No global-setup/teardown wiring; Playwright's per-test `outputDir` replaces explicit cleanup. Added 2026-04-27.
- [x] **http-auth.spec.ts** (`/basic-auth` + `/digest-auth` + `/download-secure`) — `test.use({ httpCredentials })` for context-level Basic auth, scoped per describe block (no-creds / wrong-creds / valid-creds). Negative paths assert `response.status()` from `page.goto`; valid paths assert page H1, success alert, protected file list rendering, protected download (creds persist on the sub-resource fetch, not just navigation), and single-context persistence across all three protected URLs sharing the "Restricted Area" realm. 7 TCs. `/digest-auth` is mislabeled — server actually uses Basic auth; documented as a CLAUDE.md gotcha. `admin/admin` reaches all three endpoints. `triggerDownload` extracted to `helpers/downloads.ts` and shared with `FileDownloadPage` to avoid duplication. Downloads saved to `testInfo.outputDir`. Added 2026-04-27.
- [x] **iframe.spec.ts** (`/iframe`) — `page.frameLocator('#email-subscribe')` to enter a same-origin iframe, frame-scoped locators for the embedded Bootstrap `needs-validation` subscribe form (input/button/heading/`.invalid-feedback`/`#success-message`). Covers smoke access (frame heading), EP negative paths (empty + malformed email both surface `.invalid-feedback`), happy path (valid email replaces the form's parent `#subscribe` div with `#success-message`), and a state-transition recovery flow inside the frame. 5 TCs. Targets only the email-subscribe iframe — YouTube embed is blocked by Chromium and the "TinyMCE Editor" section is not actually an iframe; both documented as a CLAUDE.md gotcha. Added 2026-04-27.
- [x] **multiple-windows.spec.ts** (`/windows`) — canonical `Promise.all([context.waitForEvent('page'), link.click()])` pattern wrapped in a POM `openNewWindow()` helper, `newPage.waitForLoadState('domcontentloaded')`, `newPage.context() === page.context()` to assert shared `BrowserContext`, `context.pages()` lifecycle assertions (1 → 2 → 1), `page.bringToFront()` to return focus, and an `openMultipleWindows(N)` helper that opens popups sequentially (parallel clicks would race the `page` event listener). 6 TCs. Opener `/windows` is one H1 + a bare `<a href="/windows/new" target="_blank">Click Here</a>` (no id/class/JS); popup `/windows/new` is same-origin and contains only an H1. No EP/BVA/decision-table TCs — no input surface; state transition (open → close → re-open) covered as TC06. Added 2026-05-11.
- [x] **hovers-tooltips.spec.ts** (`/hovers` + `/tooltips` combined in one spec) — `locator.hover()` against CSS-only `.figcaption` reveal, data-driven loops (per-user / per-button) for parallel coverage, mouse-leave via `page.mouse.move(0, 0)`, `toBeAttached/not.toBeAttached` against Bootstrap's dynamic tooltip DOM, Bootstrap's `'hover focus'` trigger semantics including click→focus retention and click-elsewhere blur. 17 TCs across two `test.describe` blocks (Hovers TC01–TC08, Tooltips TC09–TC17). Gotchas surfaced during development: (1) Bootstrap 5 strips the `title` attribute on first show, so titles must be cached *before* any hover (caught in TC17 — fixed by reading `getAttribute('title')` up front). (2) A third-party consent-management UI injects ~85 `<button role="tooltip" class="fc-help-tip">` nodes asynchronously, breaking a broad `[role="tooltip"]` selector (caught in TC15 — fixed by narrowing the POM's `getTooltip()` to `div.tooltip[role="tooltip"]`). (3) Internal eager `toBeAttached` assertions inside POM action methods trip strict mode during click-driven cross-button transitions where two tooltips are momentarily attached; callers' own auto-retrying assertions are the right place to wait. (4) Hover-reveal + click races `scrollIntoViewIfNeeded`: hovering changes layout (caption display:none → block), a later click's scroll shifts the page, mouse loses the `:hover` figure, caption hides, click times out — fixed in `hoverUser()` by `scrollIntoViewIfNeeded()` *before* `hover()`. All four documented as CLAUDE.md gotchas. Added 2026-05-11.
- [x] **key-presses.spec.ts** (`/key-presses`) — `page.keyboard.press()` for named/single keys, `page.keyboard.type()` for sequences, modifier chords via `'Shift+a'`, data-driven loops with explicit `tc` numbers per row for sequential generated titles. Page handler is `$(document).keydown` writing `"You entered: <keyboardMap[event.which]>"` to `#result`; `event.which` is the legacy jQuery keyCode (physical key) so letters always render uppercase regardless of Shift state. 22 TCs: smoke (TC01), alphanumeric BVA `a`/`z`/`0`/`9` (TC02-05), specials Enter/Tab/Escape/Space/Backspace (TC06-10), arrows (TC11-14), modifiers pressed alone Shift/Control/Alt/Meta (TC15-18), Shift+a combo demonstrating last-keydown-wins (TC19), state transition `a`→`b` with `#target.value` accumulation (TC20), typed alphanumeric `abc123` (TC21), typed unshifted specials `.,/` → "SLASH" (TC22). Gotchas surfaced and documented in CLAUDE.md: (1) `#target` lives inside a hidden `<form method="get">` — Enter while focused submits and reloads the page to `/key-presses?`, so auto-focus removed from `beforeEach` and added only to TC19-TC22 which read `#target.value`. (2) Playwright's `press('Shift+a')` chord does NOT compose uppercase characters in the input — TC19 asserts lowercase `'a'`, with comment explaining the distinction. Function keys deliberately out of scope per TODO ("event-capture display only", "no retrofits to other specs"); shifted symbols (`!`, `@`, etc.) avoided in TC22 because `event.which` is keycode not character, making the result text non-obvious. Added 2026-05-11.

## Out of scope (confirmed, not planned)

- Dynamic/wait pages: `/dynamic-loading`, `/dynamic-controls`, `/slow`, `/shifting-content`, `/disappearing-elements`, `/infinite-scroll`, `/scrollbars`
- DOM-robustness pages: `/shadowdom`, `/challenging-dom`, `/large`, `/dynamic-id`
- Visual snapshots (`toHaveScreenshot`), accessibility scans, network interception, storage/cookie inspection
- Integration between UI and API phases (phases stay independent)

## Decisions locked

- Hovers + tooltips → one combined spec, not two
- Keyboard → event-capture only, no Tab-focus retrofits in other specs
- HTTP auth → `newContext({ httpCredentials })` only (no URL-embedded credentials variant)
- File download → demonstrate `saveAs()` to a project path with cleanup
